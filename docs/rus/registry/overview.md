# Система регистров — Обзор

### Что такое регистры?

В сетевом протоколе Minecraft **регистр** (registry) — это упорядоченное отображение между человекочитаемыми именами с пространством имён (например, `minecraft:stone`) и компактными целочисленными ID (например, `1`). У каждого блока, предмета, типа сущности, биома, зачарования и десятков других категорий есть свой регистр.

Регистры решают две ключевые задачи:

1. **Эффективность сети** — вместо отправки `"minecraft:diamond_sword"` (26 байт) протокол передаёт целочисленный ID (VarInt, обычно 1–2 байта). Клиент и сервер должны использовать абсолютно одинаковое отображение имя↔ID, иначе игра сломается.
2. **Синхронизация данных** — некоторые регистры (*конфигурационные регистры*) содержат дополнительные JSON/NBT-данные, определяющие поведение записей (эффекты зачарований, свойства биомов, масштабирование урона). Эти данные отправляются клиенту на этапе Configuration.

### Типы регистров

#### Встроенные регистры (80 регистров)

Эти регистры вшиты в JAR-файл клиента и сервера Minecraft. Их отображения имя↔ID фиксированы для конкретной версии игры и не меняются в рантайме. Примеры:

| Регистр | Записей | Описание |
|---|---|---|
| `minecraft:block` | ~1060 | Все типы блоков |
| `minecraft:item` | ~1350 | Все типы предметов |
| `minecraft:entity_type` | ~130 | Все типы сущностей |
| `minecraft:fluid` | 4 | Вода, лава, их текущие формы, пустота |
| `minecraft:sound_event` | ~1300 | Все звуковые эффекты |
| `minecraft:particle_type` | ~100 | Все типы частиц |

Полный список 80 регистров с их протокольными ID берётся из `registries.json` — файла, сгенерированного встроенным генератором данных ванильного сервера Minecraft. Хранится по пути `.mcdata/1.21.4/registries.json`.

#### Конфигурационные регистры (12 регистров)

Эти регистры содержат NBT-данные, которые сервер обязан отправить клиенту на этапе Configuration. Они определяют критические для геймплея свойства:

| Регистр | Записей | Содержимое |
|---|---|---|
| `minecraft:enchantment` | 42 | Эффекты зачарований, применимые предметы, макс. уровни |
| `minecraft:damage_type` | 49 | Масштабирование урона, истощение, типы сообщений о смерти |
| `minecraft:dimension_type` | 4 | Уровни освещения, высота, окружающее освещение |
| `minecraft:worldgen/biome` | 65 | Температура, осадки, цвета тумана/неба, музыка |
| `minecraft:banner_pattern` | 43 | ID ассетов паттернов знамён |
| `minecraft:chat_type` | 7 | Шаблоны форматирования чат-сообщений |
| `minecraft:instrument` | 8 | Звуки козьих рогов |
| `minecraft:jukebox_song` | 19 | Звуки и описания музыкальных дисков |
| `minecraft:painting_variant` | 50 | Размеры картин и ID ассетов |
| `minecraft:trim_material` | 11 | Цвета материалов отделки брони |
| `minecraft:trim_pattern` | 18 | Паттерны отделки брони |
| `minecraft:wolf_variant` | 9 | Варианты текстур волков |

#### Теги

Теги — это именованные группы записей регистров. Например, тег `minecraft:enchantable/head_armor` в регистре `minecraft:item` содержит ID всех предметов, на которые можно наложить зачарования для шлема. Клиент использует теги для:

- Разрешения ссылок `HolderSet` в данных зачарований и типов урона
- Определения категорий поведения блоков/предметов (напр. `minecraft:mineable/axe`)
- Группировки биомов по признакам (напр. `minecraft:is_ocean`)

Теги отправляются пакетом `UpdateTags` на этапе Configuration. Каждый тег отображается в список целочисленных ID регистра.

### Архитектура в Vitis

#### Структура файлов

```
internal/registry/
├── registry.go          — IDRegistry: O(1) двунаправленный поиск имя↔ID
├── manager.go           — Manager: хранит все регистры, неизменяем после инициализации
├── builtin.go           — Мост: маппинг 80 имён регистров → сгенерированные []string
├── codec.go             — BuildRegistryDataPackets(): конфиг-регистры → пакеты протокола
├── tags.go              — BuildUpdateTags(): формирует пакет UpdateTags из данных тегов
├── registry_test.go     — 15+ тестов, 4 бенчмарка
├── generated/
│   ├── ids.go           — [СГЕНЕРИРОВАН] 80 регистров как []string (~6200 строк)
│   ├── config_nbt.go    — [СГЕНЕРИРОВАН] 325 предкодированных NBT-массивов
│   └── tags.go          — [СГЕНЕРИРОВАН] 541 маппинг тег→ID для 13 регистров
└── generator/
    └── main.go          — Кодогенератор (читает JSON-данные, выдаёт Go-файлы)
```

#### IDRegistry

`IDRegistry` — основная структура данных. Хранит:

- `byName map[string]int32` — имя → протокольный ID (поиск O(1))
- `byID []string` — протокольный ID → имя (поиск O(1), без аллокаций)

Бенчмарки: ~16нс имя→ID, ~6нс ID→имя, ноль аллокаций.

Неизменяем после создания. Все 92 регистра (80 встроенных + 12 конфигурационных, с частичным пересечением) хранятся в `Manager`.

#### Manager

`Manager` создаётся один раз при старте сервера через `NewManager()`. Он:

1. Загружает все 80 встроенных регистров из `generated/ids.go`
2. Загружает 12 конфигурационных регистров с предкодированным NBT из `generated/config_nbt.go`
3. Валидирует все регистры на внутреннюю согласованность

Предоставляет методы для:
- **Поиска**: `IDByName(registry, name)`, `NameByID(registry, id)`
- **Построения пакетов**: `BuildRegistryDataPackets(clientKnowsVanilla)`, `BuildUpdateTags()`

#### Оптимизация Known Packs

На этапе Configuration сервер отправляет пакет `ClientboundKnownPacks` со списком `minecraft:core 1.21.4`. Если клиент в ответе подтверждает, что знает этот пакет данных, сервер отправляет пакеты `RegistryData` с `HasData=false` для каждой записи — клиент использует свои встроенные данные вместо парсинга NBT от сервера. Это экономит трафик и избегает потенциальных расхождений сериализации.

Если клиент НЕ знает ванильный пакет (например, модифицированный клиент), сервер отправляет полные NBT-данные для каждой записи конфигурационного регистра.

#### Кодогенерация

Ни один из сгенерированных файлов не написан вручную. Генератор (`internal/registry/generator/main.go`) читает сырые файлы данных и производит исходный код на Go:

| Вход | Выход | Содержимое |
|---|---|---|
| `registries.json` | `generated/ids.go` | 80 регистров, ~6200 строк `[]string` |
| `registries.json` + `.mcdata/1.21.4/datapacks/*.json` | `generated/config_nbt.go` | 325 предкодированных `[]byte` NBT-литералов |
| `registries.json` + `.mcdata/1.21.4/tags/**/*.json` | `generated/tags.go` | 541 определение тегов как `map[string][]int32` |

Такой подход обеспечивает:
- **Ноль парсинга JSON/файлов в рантайме** — все данные вкомпилированы в бинарник
- **Типобезопасность** — никаких `map[string]interface{}` в рантайме
- **Быстрый запуск** — нет I/O, нет аллокаций сверх статических слайсов

#### Интеграция с протоколом

На этапе Configuration (`internal/session/configuration_handler.go`):

1. Сервер отправляет `ClientboundKnownPacks` с `minecraft:core 1.21.4`
2. Клиент отвечает `ServerboundKnownPacks`
3. Сервер проверяет, знает ли клиент ванильный пакет, затем:
   - Отправляет 12 пакетов `RegistryData` (с NBT или без, в зависимости от Known Packs)
   - Отправляет 1 пакет `UpdateTags` со всеми данными тегов
   - Отправляет `FinishConfiguration`
