# Обзор архитектуры

Vitis построен как многоуровневая система, где каждый уровень имеет чёткие обязанности и минимальную связанность с другими уровнями.

## Уровни

### 1. Сетевой уровень (`internal/network/`)

Самый нижний уровень. Обрабатывает TCP-соединения, фрейминг байтов и пулы I/O воркеров.

- **Listener** — принимает входящие TCP-соединения с настраиваемым лимитом подключений
- **Conn** — циклы чтения/записи на соединение с управлением дедлайнами
- **Pipeline** — расширяемая цепочка обработчиков (декодирование фреймов → диспатч в сессию)
- **BufferPool** — переиспользование буферов на базе `sync.Pool` для снижения нагрузки на GC
- **WorkerPool** — ограниченный пул горутин для обработки пакетов

### 2. Протокольный уровень (`internal/protocol/`)

Реализация проводного протокола Minecraft 1.21.4.

- **Buffer** — zero-copy буфер чтения/записи для протокольных типов (VarInt, VarLong, String, UUID, NBT и т.д.)
- **Registry** — маппинг ID пакета → конструктор с учётом версии и состояния
- **Decoder/Encoder** — декодирование фрейм → пакет и кодирование пакет → фрейм
- **States** — регистрация пакетов для Handshake, Status, Login, Configuration, Play
- **Crypto** — генерация RSA-ключей, потоковый шифр CFB8 для онлайн-режима

### 3. Уровень сессий (`internal/session/`)

Управление жизненным циклом подключённого игрока от рукопожатия до игры.

- **Session** — конечный автомат (handshake → login → configuration → play)
- **PacketRouter** — copy-on-write диспатч обработчиков по (состояние, ID пакета)
- **Manager** — потокобезопасное отслеживание сессий с поиском по ID и соединению
- **Handlers** — status, login (с онлайн-аутентификацией), configuration (синхронизация реестров), play (геймплей)

### 4. Уровень мира (`internal/world/`)

Владеет всем состоянием мира. Мутации происходят исключительно в горутине тика мира.

- **World** — центральное состояние мира: чанки, игроки, сущности, время, граница
- **Manager** — контейнер для нескольких миров (сейчас один мир)
- **Система чанков** (`chunk/`) — асинхронная загрузка/генерация с ограниченным пулом воркеров, результаты применяются на тике
- **Стриминг** (`streaming/`) — стриминг чанков для каждого игрока с очередью приоритетов по манхэттенскому расстоянию
- **Terrain** (`terrain/`) — генераторы: simplex-шум и плоский мир
- **Region** (`region/`) — чтение/запись файлов формата Anvil .mca
- **Level** (`level/`) — level.dat и сохранение данных игроков в NBT

### 5. Уровень сущностей (`internal/entity/`)

Типы сущностей с физической симуляцией.

- **Player** — расширяет LivingEntity привязкой к сессии и дистанцией обзора
- **LivingEntity** — здоровье, еда, опыт, урон, респаун
- **Physics** (`physics/`) — AABB-коллизии, гравитация, подъём на ступени, сканирование коллизий с блоками
- **Item/XPOrb/TNT** — типы сущностей со специфичным поведением на тике
- **Projectiles** (`projectile/`) — стрелы, бросаемые предметы с ray-cast обнаружением попаданий
- **Vehicles** (`vehicle/`) — лодки, вагонетки с управлением пассажирами

### 6. Уровень реестров (`internal/registry/`)

Предсобранные данные реестров Minecraft для синхронизации с клиентом.

- 78+ встроенных реестров из `registries.json`
- 12 конфигурационных реестров из датапаков (NBT-кодировка)
- Система тегов для группировки биомов/зачарований
- Zero-allocation поиск ID ↔ имя

### 7. Генерация данных (`internal/data/`)

59 генераторов кода, преобразующих файлы данных Minecraft в типобезопасный Go-код.

- Вход: PrismarineJS JSON, `registries.json`, misode/mcmeta датапаки, декомпилированный JAR сервера
- Выход: `internal/data/generated/` — ID и константы, проверяемые на этапе компиляции

## Модель потоков

```
Главная горутина        → инициализация, обработка сигналов, завершение
Горутина тик-лупа       → world.Tick() на 20 TPS
  └─ воркеры чанков     → асинхронная загрузка/генерация (ограниченный пул)
I/O воркеры сети        → циклы чтения на соединение
Пул воркеров пакетов    → декодирование + диспатч (ограниченный пул)
Запись на сессию        → асинхронная очередь исходящих на соединение
Наблюдатель конфига     → обнаружение изменений файла для горячей перезагрузки
Чтение консоли          → диспатч команд из stdin
```

## Поток данных

```
TCP клиента → цикл чтения Conn → декодирование фрейма → Pipeline
  → Session.processInboundFrame → Decoder → PacketRouter → Handler
  → Handler изменяет состояние / планирует действие в мире
  → World.Tick() применяет мутации
  → Отправка пакета → очередь исходящих Session → Encoder → цикл записи Conn → TCP клиента
```
