# Registry System — Overview

### What are registries?

In Minecraft's network protocol, a **registry** is an ordered mapping between human-readable namespaced names (e.g. `minecraft:stone`) and compact integer IDs (e.g. `1`). Every block, item, entity type, biome, enchantment, and dozens of other categories have their own registry.

Registries serve two critical purposes:

1. **Network efficiency** — instead of sending `"minecraft:diamond_sword"` (26 bytes) over the wire, the protocol sends its integer ID (a single VarInt, often 1–2 bytes). Both client and server must agree on exactly the same name↔ID mapping, or the game breaks.
2. **Data synchronization** — some registries (called *configuration registries*) carry additional JSON/NBT data that defines how entries behave (e.g. enchantment effects, biome properties, damage type scaling). This data must be sent to the client during the Configuration phase.

### Types of registries

#### Builtin registries (80 registries)

These registries are hardcoded into the Minecraft client and server JAR. Their name↔ID mappings are fixed for a given game version and never change at runtime. Examples:

| Registry | Entries | Description |
|---|---|---|
| `minecraft:block` | ~1060 | All block types |
| `minecraft:item` | ~1350 | All item types |
| `minecraft:entity_type` | ~130 | All entity types |
| `minecraft:fluid` | 4 | Water, lava, their flowing forms, empty |
| `minecraft:sound_event` | ~1300 | All sound effects |
| `minecraft:particle_type` | ~100 | All particle effects |

The full list of 80 registries with their protocol IDs comes from `registries.json` — a file generated by the vanilla Minecraft server's built-in data generator. It is stored at `.mcdata/1.21.4/registries.json`.

#### Configuration registries (12 registries)

These registries carry NBT data that the server must send to the client during the Configuration protocol phase. They define gameplay-critical properties:

| Registry | Entries | Content |
|---|---|---|
| `minecraft:enchantment` | 42 | Enchantment effects, applicable items, max levels |
| `minecraft:damage_type` | 49 | Damage scaling, exhaustion, death message types |
| `minecraft:dimension_type` | 4 | Light levels, height, ambient lighting |
| `minecraft:worldgen/biome` | 65 | Temperature, precipitation, fog/sky colors, music |
| `minecraft:banner_pattern` | 43 | Banner pattern asset IDs |
| `minecraft:chat_type` | 7 | Chat message formatting templates |
| `minecraft:instrument` | 8 | Goat horn sounds |
| `minecraft:jukebox_song` | 19 | Music disc sounds and descriptions |
| `minecraft:painting_variant` | 50 | Painting dimensions and asset IDs |
| `minecraft:trim_material` | 11 | Armor trim material colors |
| `minecraft:trim_pattern` | 18 | Armor trim patterns |
| `minecraft:wolf_variant` | 9 | Wolf texture variants |

#### Tags

Tags are named groups of registry entries. For example, the tag `minecraft:enchantable/head_armor` in the `minecraft:item` registry lists all item IDs that can receive head armor enchantments. Tags are used by the client to:

- Resolve `HolderSet` references in enchantment/damage type data
- Determine block/item behavior categories (e.g. `minecraft:mineable/axe`)
- Group biomes by features (e.g. `minecraft:is_ocean`)

Tags are sent via the `UpdateTags` packet during Configuration. Each tag maps to a list of integer registry IDs.

### Architecture in Vitis

#### File structure

```
internal/registry/
├── registry.go          — IDRegistry: O(1) bidirectional name↔ID lookup
├── manager.go           — Manager: holds all registries, immutable after init
├── builtin.go           — Bridge: maps 80 registry names → generated ID slices
├── codec.go             — BuildRegistryDataPackets(): config registry → protocol packets
├── tags.go              — BuildUpdateTags(): generates UpdateTags packet from tag data
├── registry_test.go     — 15+ tests, 4 benchmarks
├── generated/
│   ├── ids.go           — [GENERATED] 80 registries as []string slices (~6200 lines)
│   ├── config_nbt.go    — [GENERATED] 325 pre-encoded NBT byte arrays
│   └── tags.go          — [GENERATED] 541 tag→ID mappings for 13 registries
└── generator/
    └── main.go          — Code generator (reads JSON data, outputs Go files)
```

#### Data sources

All raw data lives in `.mcdata/1.21.4/`:


#### IDRegistry

`IDRegistry` is the core data structure. It stores:

- `byName map[string]int32` — name → protocol ID (O(1) lookup)
- `byID []string` — protocol ID → name (O(1) lookup, zero-alloc)

Benchmarks: ~16ns name→ID, ~6ns ID→name, zero allocations.

It is immutable after construction. All 92 registries (80 builtin + 12 config, with some overlap) are held by the `Manager`.

#### Manager

`Manager` is created once at server startup via `NewManager()`. It:

1. Loads all 80 builtin registries from `generated/ids.go`
2. Loads 12 configuration registries with pre-encoded NBT from `generated/config_nbt.go`
3. Validates all registries for internal consistency

It exposes methods for:
- **Lookup**: `IDByName(registry, name)`, `NameByID(registry, id)`
- **Packet building**: `BuildRegistryDataPackets(clientKnowsVanilla)`, `BuildUpdateTags()`

#### Known Packs optimization

During the Configuration phase, the server sends a `ClientboundKnownPacks` packet listing `minecraft:core 1.21.4`. If the client responds confirming it knows this pack, the server sends `RegistryData` packets with `HasData=false` for every entry — the client uses its own built-in data instead of parsing NBT from the server. This saves bandwidth and avoids potential serialization mismatches.

If the client does NOT know the vanilla pack (e.g. modded client), the server sends full NBT data for every configuration registry entry.

#### Code generation

None of the generated files are hand-written. The generator (`internal/registry/generator/main.go`) reads raw data files and produces Go source code:

| Input | Output | Content |
|---|---|---|
| `registries.json` | `generated/ids.go` | 80 registries, ~6200 lines of `[]string` slices |
| `registries.json` + `.mcdata/1.21.4/datapacks/*.json` | `generated/config_nbt.go` | 325 pre-encoded NBT `[]byte` literals |
| `registries.json` + `.mcdata/1.21.4/tags/**/*.json` | `generated/tags.go` | 541 tag definitions as `map[string][]int32` |

This approach means:
- **Zero runtime JSON/file parsing** — all data is compiled into the binary
- **Type-safe** — no `map[string]interface{}` at runtime
- **Fast startup** — no I/O, no allocation beyond the static slices

#### Protocol integration

During the Configuration phase (`internal/session/configuration_handler.go`):

1. Server sends `ClientboundKnownPacks` with `minecraft:core 1.21.4`
2. Client responds with `ServerboundKnownPacks`
3. Server checks if client knows vanilla, then:
   - Sends 12 `RegistryData` packets (with or without NBT based on Known Packs)
   - Sends 1 `UpdateTags` packet with all tag data
   - Sends `FinishConfiguration`
