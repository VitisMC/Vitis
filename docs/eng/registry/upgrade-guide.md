# Registry System — Upgrade Guide

This guide explains step-by-step how to update the Vitis registry system when upgrading to a new Minecraft version (e.g. from 1.21.4 to 1.22).

> **Prerequisites**: Linux or macOS, Go installed, `curl` and `python3` available, Java 21+ installed (for generating `registries.json`).

### Overview of what needs updating

When Minecraft releases a new version, three things can change:

1. **Registry IDs** — blocks, items, entities, etc. may be added, removed, or renumbered
2. **Configuration registry data** — enchantment effects, biome properties, damage types, etc. may change
3. **Tags** — tag groups (e.g. `#minecraft:enchantable/armor`) may gain or lose members

All three are handled by a single code generation pipeline. The update process is:

```
1. Generate registries.json from the new vanilla server JAR
2. Download config registry JSONs for the new version
3. Download tag JSONs for the new version
4. Run the code generator
5. Update version-specific constants
6. Test
```

### Step 1: Generate `registries.json`

#### What is `registries.json`?

This file contains all 80 builtin registry name↔ID mappings. It is generated by the vanilla Minecraft server itself using its built-in data generator.

#### Where it's stored

```
.mcdata/1.21.4/registries.json
```

For a new version, the path is `.mcdata/{version}/registries.json`, e.g. `.mcdata/1.22/registries.json`.

#### How to generate it

1. **Download the vanilla server JAR** for the target version from [Minecraft's official launcher meta](https://piston-meta.mojang.com/mc/game/version_manifest_v2.json).

   You can find the direct download URL by:
   ```bash
   curl -s https://piston-meta.mojang.com/mc/game/version_manifest_v2.json \
     | python3 -c "
   import sys, json
   manifest = json.load(sys.stdin)
   for v in manifest['versions']:
       if v['id'] == '1.22':  # ← change to target version
           print(v['url'])
           break
   "
   ```

   Or simply download from https://www.minecraft.net/en-us/download/server.

2. **Run the data generator**:
   ```bash
   mkdir -p /tmp/mc-datagen && cd /tmp/mc-datagen
   java -DbundlerMainClass="net.minecraft.data.Main" -jar server.jar --all
   ```

   This creates a `generated/` directory. The file you need is:
   ```
   generated/reports/registries.json
   ```

3. **Copy it into the project data directory**:
   ```bash
   mkdir -p /path/to/vitis/.mcdata/1.22
   cp generated/reports/registries.json \
     /path/to/vitis/.mcdata/1.22/registries.json
   ```

### Step 2: Download configuration registry JSONs

#### What are these files?

JSON files describing each entry in the 12 configuration registries (enchantments, biomes, damage types, etc.). They are converted to NBT at build time and sent to clients that don't have the vanilla data pack.

#### Source

The easiest source is the [misode/mcmeta](https://github.com/misode/mcmeta) repository on GitHub. It publishes extracted data for every Minecraft version as git tags.

Tag naming convention: `{version}-data-json` (e.g. `1.21.4-data-json`, `1.22-data-json`).

The data lives under `data/minecraft/{registry_name}/` in that tag.

#### How to download

1. **Edit `scripts/update_version.sh`**:

   Pass the target version as argument. The `REGISTRIES` array lists all datapacks to download:
   ```bash
   ./scripts/update_version.sh 1.22
   ```

   If the new version adds or removes configuration registries, update the `REGISTRIES` and `WORLDGEN_REGISTRIES` arrays:
   ```bash
   REGISTRIES=(
     "banner_pattern"
     "chat_type"
     "damage_type"
     "dimension_type"
     "enchantment"
     "instrument"
     "jukebox_song"
     "painting_variant"
     "trim_material"
     "trim_pattern"
     "wolf_variant"
   )
   ```

   > **How to know which registries exist**: Check the [Minecraft Wiki protocol page](https://minecraft.wiki/w/Java_Edition_protocol) for your version — the Configuration phase section lists all `RegistryData` registries the vanilla server sends.

2. The script downloads all JSON files into `.mcdata/{version}/datapacks/`.

#### Verifying the download

Check that each registry directory contains `.json` files:
```bash
for dir in .mcdata/1.22/datapacks/*/; do
  count=$(ls -1 "$dir"*.json 2>/dev/null | wc -l)
  echo "$(basename $dir): $count entries"
done
```

### Step 3: Download tag JSONs

#### What are tags?

Tags define named groups of registry entries. For example, `tags/item/enchantable/armor.json` lists all items that can receive armor enchantments. Tags are critical — without them, the client cannot parse enchantment data and will crash.

#### Source

Same repository: [misode/mcmeta](https://github.com/misode/mcmeta), same tag (`{version}-data-json`).

Tags live under `data/minecraft/tags/{registry_name}/` and may have subdirectories (e.g. `tags/item/enchantable/`, `tags/block/mineable/`).

#### How to download

The same `scripts/update_version.sh` handles tags. If the new version adds new tag categories, update the `TAG_REGISTRIES` array:

```bash
TAG_REGISTRIES=(
  "banner_pattern"
  "block"
  "cat_variant"
  "damage_type"
  "enchantment"
  "entity_type"
  "fluid"
  "game_event"
  "instrument"
  "item"
  "painting_variant"
  "point_of_interest_type"
)

TAG_WORLDGEN_REGISTRIES=(
  "worldgen/biome"
  "worldgen/flat_level_generator_preset"
  "worldgen/structure"
  "worldgen/world_preset"
)
```

Tags are downloaded into `.mcdata/{version}/tags/` automatically by the script.

#### Verifying

```bash
find .mcdata/1.22/tags -name '*.json' | wc -l
```

For 1.21.4, this is 562 files. Newer versions may have more.

### Step 4: Update the code generator

#### What the generator does

The generator (`internal/registry/generator/main.go`) reads three inputs and produces three Go source files:

| Input | Output file | Content |
|---|---|---|
| `registries.json` | `generated/ids.go` | All registry name↔ID mappings as `[]string` slices |
| `registries.json` + datapack JSONs | `generated/config_nbt.go` | Pre-encoded NBT bytes for config registries |
| `registries.json` + tag JSONs | `generated/tags.go` | Tag→ID mappings as `map[string][]int32` |

#### Update paths in the generator

The generator reads from `.mcdata/{version}/` using the `-version` flag:

```bash
go run ./internal/registry/generator/ -version 1.22
```

No path changes needed — the generator uses `.mcdata/{version}/` automatically.

#### Update config registry list

If Minecraft added or removed configuration registries, update the `configRegistries` map in the generator:

```go
var configRegistries = map[string]string{
    "minecraft:banner_pattern":   "banner_pattern",
    "minecraft:chat_type":        "chat_type",
    "minecraft:damage_type":      "damage_type",
    "minecraft:dimension_type":   "dimension_type",
    "minecraft:enchantment":      "enchantment",
    "minecraft:instrument":       "instrument",
    "minecraft:jukebox_song":     "jukebox_song",
    "minecraft:painting_variant": "painting_variant",
    "minecraft:trim_material":    "trim_material",
    "minecraft:trim_pattern":     "trim_pattern",
    "minecraft:wolf_variant":     "wolf_variant",
    "minecraft:worldgen/biome":   "worldgen/biome",
}
```

#### Update NBT type hints (if needed)

Some JSON fields need explicit NBT type annotations because JSON has no distinction between float/double or int/long:

```go
var doubleFields = map[string]bool{
    "coordinate_scale": true,
    "offset":           true,
}

var longFields = map[string]bool{
    "fixed_time": true,
}
```

#### Run the generator

```bash
go run ./internal/registry/generator/ -version 1.22
```

### Step 5: Update `builtin.go`

If the new Minecraft version added or removed builtin registries, update the `builtinIDMap()` function in `internal/registry/builtin.go`.

Each entry maps a registry name to a generated `[]string` variable:

```go
"minecraft:block": generated.Block,
"minecraft:item":  generated.Item,
```

The variable names are auto-generated from registry names. Convention:
- `minecraft:block` → `generated.Block`
- `minecraft:entity_type` → `generated.EntityType`
- `minecraft:worldgen/biome_source` → `generated.WorldgenBiomeSource`

Check the top of `generated/ids.go` after regeneration to find the exact variable names.

### Step 6: Update version constants

#### Known Packs version

In `internal/session/configuration_handler.go`, update the Known Packs version string:

```go
knownPacks := &cfgpacket.ClientboundKnownPacks{
    Packs: []cfgpacket.KnownPack{
        {Namespace: "minecraft", ID: "core", Version: "1.22"},  // ← update
    },
}
```

#### Protocol version

Update the protocol version number wherever it's defined in the project. Check the [Minecraft Wiki protocol version list](https://minecraft.wiki/w/Java_Edition_protocol_version) for the correct number.

### Step 7: Build and test

#### Compile

```bash
go build ./...
```

Fix any compilation errors. Common issues:
- New registries in `registries.json` need corresponding entries in `builtin.go`
- Removed registries need their entries deleted from `builtin.go`

#### Run tests

```bash
go test ./internal/registry/ -v -count=1
```

#### Connect a client

Start the server and connect with a vanilla Minecraft client of the target version:

```bash
go run ./cmd/vitis -config configs/vitis.yaml
```

Check the server logs for:
```
registry manager: 92 registries, 12 config registries
session X: serverbound_known_packs count=1 vanilla=true
session X: sent 12 registry_data packets (knownVanilla=true)
session X: sent update_tags
session X: sent finish_configuration
session X: acknowledge_finish_configuration, entering play
```

If the client disconnects with registry errors, check:
1. Are all config registry entries present and correctly NBT-encoded?
2. Are all required tags (especially item tags like `enchantable/*`) being sent?
3. Are the NBT type hints (`doubleFields`, `longFields`) correct for new/changed fields?

### Quick reference: data sources

| Data | Source | How to get |
|---|---|---|
| `registries.json` | Vanilla Minecraft server JAR | `java -DbundlerMainClass="net.minecraft.data.Main" -jar server.jar --all` → `generated/reports/registries.json` |
| Config registry JSONs | [misode/mcmeta](https://github.com/misode/mcmeta) tag `{version}-data-json` | `scripts/update_version.sh` or browse `data/minecraft/{registry}/` |
| Tag JSONs | [misode/mcmeta](https://github.com/misode/mcmeta) tag `{version}-data-json` | `scripts/update_version.sh` or browse `data/minecraft/tags/{registry}/` |

Alternative sources:
- [PrismarineJS/minecraft-data](https://github.com/PrismarineJS/minecraft-data/tree/master/data/pc/) — community-maintained, has data for many versions
- Running the vanilla server data generator (same as for `registries.json`) — the `generated/data/minecraft/` directory contains both datapack entries and tags

### Troubleshooting

#### Client crashes with "Failed to load registries"

The client could not parse some registry entries. Common causes:
- Missing or malformed NBT data in a config registry entry
- Wrong NBT type for a numeric field (e.g. sending `Float` where `Double` is expected)

**Fix**: Check `doubleFields` / `longFields` in the generator. Compare the NBT output against a vanilla server capture.

#### Client crashes with "Failed to parse local data"

The client is using local (built-in) data (Known Packs mode, `HasData=false`) but cannot resolve some references. This means **tags are missing**.

**Fix**: Ensure all tag registries are included in `TAG_REGISTRIES` / `TAG_WORLDGEN_REGISTRIES` in the fetch script, and that the generator produces tag data for them.

#### "X registries" count doesn't match expectations

The total registry count is builtin (80) + config-only registries. Some config registries (like `minecraft:enchantment`) are ALSO in `registries.json` as builtin registries, so they overlap. The `Manager` deduplicates them.

#### Generator fails with "read datapack dir" error

The JSON files for a config registry are missing. Re-run `scripts/update_version.sh` and verify the files were downloaded.

#### Generator fails with "parse JSON" error

A downloaded JSON file is malformed (possibly a GitHub API error response). Delete the `.mcdata/` directory and re-run `scripts/update_version.sh`. If using GitHub API, be aware of rate limits — consider using a GitHub token.
